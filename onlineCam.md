# Камеры в браузере (feature/browser-cameras)

Переход с захвата USB-камер на **бэкенде** (OpenCV) на **`getUserMedia`** во **фронтенде**. Бэкенд остаётся stateless-сервисом цвета: `analyze-rgb`, `match`, дефолтная палитра из `base_colors.json`.

---

## Что изменилось

| Было | Стало |
|------|--------|
| OpenCV читает камеры на машине с API | Браузер запрашивает камеры у пользователя |
| `/api/snapshot`, `/api/color`, `/api/cameras`, ROI на бэке | Нет; ROI и поток только в SPA |
| JPEG polling ~10/s | `<video>` + canvas для цвета и захвата |
| Индексы камер 0, 1 | `deviceId` из `MediaDevices` |

---

## Плюсы

1. **Публичный веб** — каждый пользователь работает со своими камерами; серверу не нужны USB и OpenCV.
2. **Приватность** — видео не уходит на сервер; на API уходят только RGB (и опционально снимок не отправляется).
3. **Масштабирование** — нет фоновых потоков и JPEG на каждого клиента.
4. **Нет общего состояния** — ROI и захват изолированы по вкладке/пользователю.
5. **Проще деплой** — статика + лёгкий Python API без драйверов камер.
6. **HTTPS / localhost** — стандартная модель WebRTC/getUserMedia.

---

## Минусы

1. **Две камеры сразу** — не на всех ОС/браузерах стабильно; может понадобиться одна камера и переключение ролей или два `deviceId` вручную.
2. **Разрешение и цвет** — браузер выбирает профиль устройства; авто-WB/HDR влияют на краску (как и раньше, но меньше контроля, чем у OpenCV).
3. **Первый доступ** — нужен жест пользователя (разрешение камеры); без HTTPS на «чужом» домене getUserMedia может быть заблокирован.
4. **Нет удалённого стенда** — нельзя смотреть камеры в мастерской через сервер; только локальный браузер у станка.
5. **Сглаживание live** — EMA была на бэке; на фронте нужно реализовать отдельно или жить с дрожанием.
6. **Список устройств** — до разрешения камеры метки устройств могут быть пустыми (`enumerateDevices`).
7. **Отказ от mock-видео бэка** — без камеры UI показывает ошибку/подсказку, а не синтетический поток.

---

## Риски и митигация

| Риск | Митигация |
|------|-----------|
| Одна камера выбрана для обеих ролей | Предупреждение в UI; разные `deviceId` |
| Размер кадра ≠ 1280×720 | ROI в координатах `videoWidth`×`videoHeight` |
| Бэкенд недоступен | Цвета из canvas локально невозможны для Lab без API — показать offline для match |
| Safari / mobile | Тестировать constraints; fallback на одну камеру |

---

## API бэкенда после перехода

- `GET /api/health`
- `GET /api/config` — дефолты ROI/кадра (без камер)
- `POST /api/analyze-rgb`
- `GET /api/match?targetR=…` (override цветов с клиента)
- `GET /api/base-colors` — чтение дефолтной палитры (read-only файл)

Не используется: `/api/snapshot`, `/api/color`, `/api/cameras`, `/api/roi`, `/stream`.

## Дальше (ROADMAP)

- Палитра и ROI только на клиенте (`localStorage`, import JSON).
- `POST /api/match` с телом `{ target, palette, bases }` без привязки к файлу на диске.
- Опциональный stateless `POST /sample-color` (image+roi), если не хотите дублировать median в TS.
